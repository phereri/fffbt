## What this repo is

Production-grade **Instagram Trial Reels posting automation** for a fleet of
physical Android devices. Stack: Python 3.10+ (asyncio), Supabase/Postgres as
source of truth, **MobileRun** AI agent that drives each device by natural-language
goal, **GenFarmer** (Windows app) for device fingerprint/identity, GenRouter for
proxies, 5sim.net (planned) for phone verification. ShopAIKey is the LLM router
the posting path uses (`ANTHROPIC_BASE_URL=https://api.shopaikey.com`).

Key existing pieces:
- `src/worker/agent_runner/` — wraps MobileRun `MobileAgent` (factory in
  `mobilerun_agent_runner.py`, goal templates in `goal.py`, result mapping in
  `result.py`). Agent runs as ONE atomic run; output is a Pydantic model.
- `src/worker/tools/device.py:device_summary()` — reads device props over ADB.
- `src/scheduler/` — state machine + async launcher (parallel jobs).
- `docs/research/genfarmer-*.md`, `mobilerun-real-repo-task-map.md` — research.
- `config/mobilerun/` — MobileRun config + Instagram AppCards.

## The NEW task: Instagram account auto-registration (авторег)

Build a tool that picks a free Android device and registers a fresh Instagram
account on it, fully agent-driven, with human-like delays, scalable to hundreds
of devices in parallel. Phone verification via **5sim.net**. Agent invents its
own username/password each time. Output: a **CSV** with everything needed to use
the account (username, password, phone, full device fingerprint + serials).

Current phase: **local testing on an Android Studio emulator** (no GenFarmer /
proxies / real devices available yet). Flow is developed interactively — the
agent decides what to do on screen and, when it hits an unexpected screen, pauses
and asks the operator, saving all trajectories.

## Design decisions already made (brainstorm, approved by user)

1. **Standalone lightweight tool** in `src/registration/` — does NOT touch
   Supabase / state-machine yet. Reuses only `agent_runner` + `tools/device.py`.
   Layout:
   ```
   src/registration/
     cli.py         # register --device-serial <S> [--country ...] [--csv accounts.csv]
     five_sim.py    # 5sim.net client: buy_number, get_code(poll), finish, cancel, balance
     tools.py       # custom MobileRun tools: buy_phone_number(), get_sms_code(), ask_operator()
     goal.py        # build_registration_goal() — IG signup goal (analog of build_trial_reel_goal)
     identity.py    # fallback identity generation (name/birthday) if needed
     fingerprint.py # device fingerprint+serial snapshot (wraps device_summary + raw getprop dump)
     output.py      # atomic append row to CSV
     result.py      # Pydantic RegistrationResult
   ```

2. **Agent owns 5sim** via custom tools `buy_phone_number(country)` and
   `get_sms_code()` — the agent decides when to call them. Python orchestrator
   does NOT pre-own the 5sim lifecycle; the tools wrap the 5sim client and own
   the money guardrails (timeout + auto-cancel; `finish` on success, `ban`/`cancel`
   on failure). `FIVESIM_API_KEY` from env.

3. **Real pause/resume** is implemented as a **blocking custom tool**
   `ask_operator(question)`: on call it captures screenshot + UI dump to artifacts,
   prints the question to the terminal, blocks on stdin, returns the operator's
   answer to the agent. No MobileRun run-loop changes needed — same mechanism as
   the 5sim tools. (Existing `tap_share_and_confirm` already blocks ~22s polling,
   so blocking tools are fine.)

4. **MobileRun supports custom tools** — confirmed: the "real repo" registers
   them via `CUSTOM_TOOL_SPECS` / `farm.tools.build_custom_tools()` at agent
   construction (`device_summary`, `prepare_video_for_android`, etc. are exactly
   such tools). The FFFBT wrapper just doesn't wire `tools=` yet.
   **Single surgical change outside `registration/`:** add an optional `tools`
   field to `AgentFactoryRequest` in `mobilerun_agent_runner.py` and pass it into
   `MobileAgent(... tools=...)` in `mobilerun_agent_factory`. Default empty → the
   posting path is unchanged. NOTE: `mobilerun` package is NOT installed locally
   (lazy import; tests mock it) — installing it is a prerequisite for live runs.

5. **Agent invents** username, password, full name, birthday (18+).
   `RegistrationResult` (Pydantic) returns: `success, username, password,
   full_name, birthday, phone_number, fivesim_order_id, failure_reason, notes`.

## Device identity / GenFarmer findings (important)

- **GenFarmer `ChangeDevice` is a black box.** No per-field control (the "Manual"
  radio is disabled, `genfarmer-renderer-findings.md §3`); it randomly generates a
  whole new identity. Scope (per-app vs per-device) is unconfirmed. It can be
  hazardous: on a Tailscale-only phone it may kill all networking
  (`genfarmer-changeinfo-hazard.md`).
- **`ChangeDevice` is NOT in the GenFarmer REST API** (`genfarmer-api.md §2.5,
  §4.3`). The only programmatic surface is localhost REST `127.0.0.1:55554` for
  Automation Apps / Tasks / Runs. ChangeDevice / Backup / ADB / Screenshot are
  UI-only desktop features. Two unverified ways to trigger it programmatically:
  (a) as an **Automation-graph node** run via REST (`POST /automation/runs` →
  `PUT /runs/:id/run` → poll `storages`) — the hazard doc's "task graph node"
  hint suggests this exists; status likely/unknown; (b) Windows UI automation.

- **Chosen approach: rotate-then-capture, behind an interface.**
  ```
  DeviceIdentityRotator:  rotate(device) -> None
    NoopRotator           # local/emulator: no-op (capture-only) — used now
    GenFarmerAutoRotator  # fleet: trigger ChangeDevice Automation Run via REST,
                          #        THEN verify ADB reachable (hazard) with timeout
  ```
  Flow: `rotate(device)` → verify reachable → **snapshot fingerprint (always)** →
  register. Local = `NoopRotator`; fleet = `GenFarmerAutoRotator` (TODO, marked
  "verify on real farm"; this is where OQ-1 / which fields actually move gets
  answered). User chose to rotate ChangeDevice before each registration on the
  real fleet; locally we degrade to capture-only by flag.

- **We never SET fingerprint fields — we READ them** with ADB at registration
  time. `fingerprint.py` captures the full set below AND dumps raw `getprop` to an
  artifact (`raw_getprop_path`) so nothing is lost if GenFarmer touches an
  unlisted prop.

## CSV schema (`accounts.csv`)

```
username, password, full_name, birthday,
phone_number, phone_country, fivesim_order_id,
registered_at,
device_adb_serial, device_genfarmer_id, device_connection_type,
fp_model, fp_brand, fp_manufacturer, fp_product_name, fp_device,
fp_build_fingerprint, fp_build_id, fp_android_version, fp_sdk,
fp_serialno, fp_android_id, fp_gaid, fp_imei, fp_imsi, fp_boot_id,
fp_wifi_mac, fp_ip, fp_screen_w, fp_screen_h, fp_density,
fp_locale, fp_timezone, fp_carrier, fp_carrier_numeric,
raw_getprop_path, trajectory_path, status
```

Capture commands (from GenFarmer docs, all changeable-by-ChangeDevice surfaces):
- getprop: `ro.product.model/brand/manufacturer/name/device`,
  `ro.build.fingerprint/id`, `ro.build.version.release/sdk`, `ro.serialno`,
  `ro.ril.imei`, `gsm.sim.imsi`, `gsm.sim.operator.alpha/numeric`,
  `persist.sys.locale`, `persist.sys.timezone`
- settings: `settings get secure android_id`, `settings get secure advertising_id`
- other: `/proc/sys/kernel/random/boot_id`, `cat /sys/class/net/wlan0/address`,
  `wm size`, `wm density`, `adb devices` (transport serial)
- GenFarmer-only (empty locally): `devices.current_device_id`, `connection_type`

## Local testing prerequisites & risks

- **M0 prereqs:** install `mobilerun` package locally (NOT in venv — blocker);
  emulator visible in `adb devices` with Instagram installed; `FIVESIM_API_KEY`
  set + `five_sim.py` smoke-tested standalone (buy/cancel one number).
- **Risk:** Instagram aggressively flags emulators / datacenter IPs → registration
  may hit challenge/ban on the emulator. Acceptable for FLOW DEVELOPMENT (we map
  trajectories + `ask_operator` points); real success needs real devices + proxies.
- **Risk:** Portal IME (`com.mobilerun.portal`) is absent on a bare emulator →
  text input falls back to ADB keyboard (fine for username/password/code).
